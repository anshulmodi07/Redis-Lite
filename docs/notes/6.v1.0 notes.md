This documentation outlines the transition of the Redis clone to **V1.0**. This milestone marks the definitive shift from a simple text-based toy project to a legitimate database clone that can communicate directly with official client applications like `redis-cli` and industrial Redis libraries.

The core objective of V1.0 is to layer a **RESP2 (REdis Serialization Protocol version 2) Decoder** on top of the networking and dispatch layers established in V0.1 and V0.2, without tearing down the internal execution mechanics.

Here is an exhaustive architectural and structural analysis of Redis V1.0.

---

## 1. The Core Paradigm Shift: Adopting RESP2

In V0.2, you built an argument-vector tokenizer (`vector<string> argv`) designed to handle custom quotes and backslash escapes. While highly educational, official Redis clients do not communicate using arbitrary text configurations. They communicate using a highly explicit binary-safe framing protocol called RESP.

### The Mechanics of a RESP2 Command Array

#### RESP's Solution

RESP never guesses.

Instead:

```text
I will tell you EXACTLY how many bytes follow.
```

Example:

```text
$11
hello world
```

The parser doesn't search for spaces.

The parser doesn't search for quotes.

The parser simply reads:

```text
11 bytes
```

and stops.

---

Under the RESP protocol, every multi-argument command is transmitted across the network socket as an **Array of Bulk Strings**.

Let's dissect exactly what happens on the wire when a standard client sends `SET foo bar`:

```text
*3\r\n$3\r\nSET\r\n$3\r\nfoo\r\n$3\r\nbar\r\n

```

* **`*3\r\n`** $\rightarrow$ Declares an **Array** containing exactly 3 elements. The `\r\n` (CRLF) terminates the length indicator.
* **`$3\r\nSET\r\n`** $\rightarrow$ Declares the first element as a **Bulk String** containing exactly 3 bytes of payload (`SET`), followed by a terminating `\r\n`.
* **`$3\r\nfoo\r\n`** $\rightarrow$ Declares the second element as a Bulk String containing 3 bytes of payload (`foo`).
* **`$3\r\nbar\r\n`** $\rightarrow$ Declares the third element as a Bulk String containing 3 bytes of payload (`bar`).

### Why RESP is "Binary Safe"

Unlike V0.2 text tokenization, which had to inspect every byte for quotes or spaces to isolate arguments, RESP uses **explicit data length framing**. Because the parser reads the raw length digit first (e.g., `$11`), it knows exactly how many bytes to grab from the memory buffer next. The payload string can contain spaces, binary characters, control bytes, or raw graphics, and the parser will never confuse them for token delimiters.

## 2. Structural Isolation: The `feed()` / `tryParse()` Flow

V1.0 introduces a dedicated state tracking parser module encapsulated in `resp.h` and `resp.cpp`. Every connected client is assigned its own dedicated instance of a `RespParser` class.

To correctly interface with partial or fragmented TCP transfers, the parser splits stream processing into a clean two-step execution pattern:

```text
       Raw TCP Socket Read (server.cpp)
                    │
                    │ bytes received from recv()
                    ▼
          ┌───────────────────┐
          │ parser.feed(...)  │ ──> Appends raw bytes into internal buffer
          └───────────────────┘
                    │
                    ▼
          ┌───────────────────┐
          │ parser.tryParse() │ ──> Attempts to parse complete frame
          └───────────────────┘
                    │
         ┌──────────┴──────────┐
         ▼                     ▼
[ Complete Frame found ]    [ Incomplete Frame ]
   │                           │
   ├─► Extracts argv vector    └─► Leaves buffer intact;
   ├─► Consumes matching bytes     waits for next feed()
   ▼
Executes dispatch()

```

### Deferring Consumption

A crucial defense mechanism implemented in `RespParser` is **deferred consumption**. If the parser inspects the buffer and detects that it has received a partial command (e.g., it parsed `*3\r\n$3\r\nSET\r\n` but the socket hasn't delivered the remaining keys yet), `tryParse()` terminates instantly and returns an incomplete indicator. The internal buffer pointers are **not** moved. The bytes are consumed only *after* the entire command array validation succeeds.

## 3. High-Performance Network Pipelining

Because TCP streams are arbitrary, a single `recv()` call might pull down hundreds of bytes containing multiple packed commands simultaneously. This scenario is known as **Network Pipelining**.

### The V1.0 Pipeline Loop

Rather than processing a single packet and clearing the buffer, `server.cpp` now wraps the parser execution path inside an iterative loop:

```cpp
// Pseudocode inside the client thread worker loop
parser.feed(socket_buffer, bytes_read);

std::vector<std::string> argv;
while (parser.tryParse(argv)) {
    std::string response = dispatch(argv, db, db_mutex);
    sendAll(client_socket, response);
    argv.clear(); // Reset vector container for the next pipelined command
}

```

If a performance-tuned client application shoots three commands over the wire in a single network transmission packet, the `while(parser.tryParse())` loop will execute three consecutive times back-to-back, fully parsing, evaluating, and returning results sequentially without incurring additional I/O blocking delays.

---

## 4. Architectural Cleanliness & Backward Compatibility

### Preserving the Core Signature

A hallmark of solid software engineering showcased in V1.0 is that the core command routing table inside `parser.cpp` did not need a rewrite. Because V0.2 established the internal convention of executing operations through a standardized `vector<string> argv` signature, the new `RespParser` simply converts the extracted RESP components into that exact array structure. The command engine layer remains blind to whether the commands originally arrived as raw text or structured RESP frames.

### Inline Text Fallback Mode

To prevent breaking existing automated test infrastructure (`test_v0_1.py`, `test_v0_2.py`) and to retain support for testing via manual command terminals (like Telnet or Netcat), the `RespParser` features an **inline fallback mode**.

* If `tryParse()` looks at the start of the buffer and does not find a RESP array character (`*`), it diverts processing execution to the legacy text tokenizer.
* This design aligns closely with native production Redis architectures, which allow developers to send quick, plaintext commands to query status values directly.

---

## 5. Protocol Constraints & Next Horizons

Despite achieving native command processing validation parity with production Redis instances, version 1.0 retains explicit development constraints designed to prevent scope creep:

* **Asymmetric Protocol Serialization:** While the server now successfully accepts incoming RESP arrays, **outgoing responses are still formatted using legacy plain text**. If you attach an un-modified `redis-cli` instance, you can successfully mutate keys, but the terminal interface will likely throw formatting errors because it expects structured RESP tokens back (like `+OK\r\n` or `$5\r\nvalue\r\n`) rather than naked strings.
* **Input Restrictions:** The current parsing structure strictly enforces that inputs must be formatted as flat arrays of bulk strings. Any alternate RESP types (such as primitive Null Arrays or Null Bulk Strings) sent as primary database entry vectors will trigger an immediate protocol error rejection.
* **Concurrency Limitations:** The execution layer remains tied to a single blocking worker thread per connected client socket file descriptor, leaving asynchronous non-blocking performance scaling (such as `epoll` or an intensive single-threaded loop framework) for future development phases.
