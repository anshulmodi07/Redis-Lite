# 1. Introduction

Redis Clone V0.2 focuses on the Application Layer.

While V0.1 concentrated on:

* TCP stream handling
* Partial packet processing
* Multi-client support
* Thread safety

V0.2 completely redesigns how commands are parsed and executed.

The primary objective is to create a clean command-processing architecture capable of supporting future Redis-compatible protocols.

---

# 2. Goals of V0.2

The major objectives are:

* Support values containing spaces
* Support quoted strings
* Support escaped characters
* Separate networking from parsing
* Introduce structured command dispatching
* Implement argument validation
* Create a RESP-ready architecture

---

# 3. Problems with V0.1

V0.1 relied on simple whitespace splitting.

Example:

```text
SET message hello world
```

Would incorrectly produce:

```cpp
{
    "SET",
    "message",
    "hello",
    "world"
}
```

Problems:

* No support for spaces inside values
* No quoted strings
* No escape sequences
* Networking and parsing tightly coupled
* Difficult future protocol upgrades

---

# 4. Architectural Redesign

## 4.1 Separation of Concerns

V0.2 separates:

### Networking Layer

Responsible for:

* Accepting clients
* Reading socket data
* Sending responses

### Parsing Layer

Responsible for:

* Tokenization
* Quote handling
* Escape processing

### Dispatch Layer

Responsible for:

* Command validation
* Command routing
* Database operations

---

## 4.2 New Processing Pipeline

```text
Client
   │
   ▼
TCP Socket
   │
   ▼
server.cpp
   │
   ▼
tokenize()
   │
   ▼
vector<string> argv
   │
   ▼
dispatch()
   │
   ▼
Command Handler
   │
   ▼
Database
```

---

## 4.3 Module Responsibilities

### server.cpp

Handles:

* Network I/O
* Client connections
* Response transmission

### parser.cpp

Handles:

* String parsing
* Quote processing
* Escape processing
* Argument generation

### dispatch()

Handles:

* Validation
* Routing
* Command execution

---

# 5. Argument Vector (argv) Design

## 5.1 Why argv?

The parser converts raw text into:

```cpp
vector<string> argv;
```

This becomes the universal representation of commands.

---

## 5.2 Internal Representation

Input:

```text
SET bio "hello world"
```

Output:

```cpp
{
    "SET",
    "bio",
    "hello world"
}
```

---

## 5.3 Industry Usage

The argv model is used extensively by:

* Redis
* Linux shells
* Docker
* Git
* PostgreSQL clients

The parser's job is simply:

```text
Raw Input
    ↓
argv[]
```

---

# 6. Advanced String Tokenization

## 6.1 Tokenization Overview

Tokenization converts raw command text into structured arguments.

Input:

```text
SET key value
```

Output:

```cpp
{
    "SET",
    "key",
    "value"
}
```

---

## 6.2 Quoted Strings

Quoted strings allow spaces inside values.

Input:

```text
SET message "hello world"
```

Output:

```cpp
{
    "SET",
    "message",
    "hello world"
}
```

Inside quotes:

* Spaces lose delimiter behavior
* Entire content becomes one token

---

## 6.3 Escape Character Processing

Escape character:

```text
\
```

Special handling:

```text
\" -> "
\\ -> \
```

Example:

```text
SET pattern "hi \"you\""
```

Produces:

```cpp
{
    "SET",
    "pattern",
    "hi \"you\""
}
```

Stored value:

```text
hi "you"
```

---

## 6.4 Error Handling

Invalid input:

```text
SET key "unterminated
```

Response:

```text
ERR unterminated quoted string
```

Benefits:

* No crashes
* No undefined behavior
* Graceful recovery

---

## 6.5 Complexity Analysis

Tokenizer scans once.

```cpp
for(char c : line)
```

Time Complexity:

```text
O(n)
```

Where:

```text
n = command length
```

No nested loops.

No backtracking.

---

# 7. Finite State Machine Parser

The tokenizer behaves as a finite state machine.

---

## 7.1 NORMAL State

Default state.

Spaces terminate tokens.

Example:

```text
SET key value
```

---

## 7.2 IN_QUOTES State

Entered after:

```text
"
```

Spaces become ordinary characters.

Example:

```text
"hello world"
```

---

## 7.3 ESCAPED State

Entered after:

```text
\
```

Next character treated literally.

Example:

```text
\"
```

becomes:

```text
"
```

---

# 8. Command Normalization

## 8.1 Case Insensitivity

Redis commands are case-insensitive.

Examples:

```text
set
SET
SeT
```

All become:

```text
SET
```

---

## 8.2 Canonical Representation

Only:

```cpp
argv[0]
```

is uppercased.

Example:

Input:

```text
SET username Devam
```

Result:

```cpp
{
    "SET",
    "username",
    "Devam"
}
```

Values remain unchanged.

---

# 9. Command Dispatch Engine

## 9.1 Dispatcher Responsibilities

The dispatcher:

* Receives argv
* Validates arguments
* Selects handler
* Returns response

---

## 9.2 Routing Strategy

Possible implementation:

```cpp
unordered_map<string, HandlerFunction>
```

Example:

```cpp
handlers["GET"] = handleGet;
handlers["SET"] = handleSet;
handlers["DEL"] = handleDel;
```

---

## 9.3 Command Lookup

Flow:

```text
argv
 │
 ▼
Command?
 │
 ├── GET
 ├── SET
 ├── DEL
 └── UNKNOWN
```

---

# 10. Arity Validation

## 10.1 What is Arity?

Arity = expected argument count.

Examples:

| Command | Expected Tokens |
| ------- | --------------- |
| GET     | 2               |
| SET     | 3               |
| DEL     | 2               |

---

## 10.2 Defensive Programming

Before:

```cpp
argv[1]
```

The dispatcher checks:

```cpp
argv.size()
```

This prevents:

* Out-of-bounds access
* Crashes
* Undefined behavior

---

## 10.3 Redis-Compatible Errors

Example:

```text
GET key extra
```

Response:

```text
ERR wrong number of arguments for 'GET' command
```

Benefits:

* Easier debugging
* Redis-like behavior
* Better client compatibility

---

# 11. Database Interaction Model

## 11.1 Mutex Ownership

Database ownership remains in:

```cpp
server.cpp
```

Parser never owns state.

---

## 11.2 Reference Injection

Dispatcher receives:

```cpp
dispatch(argv, db, db_mutex)
```

Advantages:

* Explicit dependencies
* Easier testing
* Better modularity

---

## 11.3 Locking Strategy

Example:

```cpp
lock_guard<mutex> lock(db_mutex);
```

Lock acquired only when required.

Benefits:

* Reduced contention
* Better concurrency
* Simpler thread safety

---

# 12. Why V0.2 Matters

## 12.1 Abstraction Boundaries

Everything below dispatch only depends on:

```cpp
vector<string> argv
```

Not on:

* TCP
* Plain text
* RESP

---

## 12.2 Dependency Inversion

Command execution no longer depends on:

```text
Text Protocol
```

It depends on:

```text
argv
```

A higher-level abstraction.

---

## 12.3 Future Protocol Upgrades

Any parser capable of generating:

```cpp
vector<string>
```

can use the existing command engine.

---

# 13. Path to RESP (V1.0)

Current:

```text
SET key value
```

↓

```cpp
{
    "SET",
    "key",
    "value"
}
```

Future RESP:

```text
*3
$3
SET
$3
key
$5
value
```

↓

```cpp
{
    "SET",
    "key",
    "value"
}
```

Dispatch remains unchanged.

Only the parser is replaced.

This is the primary architectural achievement of V0.2.

---

# 14. Current Limitations

## Threading Model

Current:

```text
1 Client → 1 Thread
```

Not scalable to thousands of clients.

Future solutions:

* epoll
* poll
* select
* io_uring

---

## No Persistence

Database lives entirely in RAM.

When process exits:

```text
All data lost
```

Future possibilities:

* RDB snapshots
* Append Only Files (AOF)

---

## Simplified Quoting

Supported:

```text
"hello world"
\"quoted\"
```

Not supported:

```bash
$VARIABLE
$(command)
`backticks`
pipes
single-quote shell semantics
```

The parser is intentionally lightweight.

# 16. Key Takeaways

### V0.1

Focused on transport-layer reliability:

* TCP buffering
* Multi-client handling
* Thread safety

### V0.2

Focused on application-layer architecture:

* Argument tokenization
* Quoted string support
* Escape processing
* Command normalization
* Arity validation
* Dispatch abstraction

### Biggest Achievement

The command engine now depends only on:

```cpp
vector<string> argv
```

This creates a stable abstraction boundary that allows the text protocol to be completely replaced by RESP in V1.0 without modifying command execution logic.

V0.2 transforms the project from a basic socket-based key-value server into a properly layered database architecture.
