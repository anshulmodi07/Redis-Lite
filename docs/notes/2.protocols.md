# Network Protocols in a Redis Clone

When building a Redis clone, you do not need to implement lower-level networking layers (Layers 1 through 3) from scratch. The operating system's network stack handles those details for you. 

As a software engineer writing the server code, your application will interact **directly** with three specific protocols across the Transport, Presentation, and Application layers.

---

## 1. TCP (Transmission Control Protocol) — Layer 4
TCP is the foundational transport protocol you will write your socket code against. Redis relies exclusively on TCP because a database requires absolute data integrity; UDP's "fire-and-forget" model is completely unsuited for reliable state management.

### Direct Implications for Your Code:
* **Byte Stream Management:** As highlighted in your `Redis_v0.1` architecture, TCP does not preserve application layer message boundaries. You must write custom **receive buffers** to continuously ingest raw bytes and scan them for your protocol's delimiter (like a newline character `\n` or `\r\n`) to reconstruct distinct commands.
* **Connection Lifecycle Management:** You will handle the standard OS socket API lifecycle: initializing a listening socket, binding it to a port, executing `accept()` to spawn client file descriptors, and tracking active connections inside a thread pool or an event loop.
* **Socket Tuning via Flags:** You will directly alter TCP behaviors using the `setsockopt()` system call. Crucial adjustments include setting `SO_REUSEADDR` to reclaim ports bound in `TIME_WAIT` during rapid server restarts, and toggling `TCP_NODELAY` to disable Nagle's algorithm, forcing packets out immediately to minimize command latency.

---

## 2. RESP (REdis Serialization Protocol) — Layer 7
While TCP moves raw bytes over wires, **RESP** defines exactly what those bytes mean. This is the custom application-layer protocol designed specifically for Redis. While your `v0.1` iteration relied on basic plain-text string parsing, a complete Redis clone must accurately serialize and deserialize RESP payloads.

### Direct Implications for Your Code:
You will build a serialization engine in your clone to dynamically parse data based on explicit RESP type prefixes:
* **Simple Strings:** Prefixed with a plus sign (e.g., `+OK\r\n`). Used for lightweight, non-binary data.
* **Errors:** Prefixed with a minus sign (e.g., `-ERR unknown command\r\n`). This structure maps cleanly to exceptions in your application.
* **Integers:** Prefixed with a colon (e.g., `:1000\r\n`).
* **Bulk Strings:** Prefixed with a dollar sign and a length indicator (e.g., `$5\r\nhello\r\n`). These are binary-safe because the explicit length tells the parser exactly how many bytes to read, ignoring embedded special characters.
* **Arrays:** Prefixed with an asterisk (e.g., `*2\r\n$3\r\nGET\r\n$4\r\nname\r\n`). Every multi-argument command sent to your server will arrive formatted as a RESP Array.

---

## 3. TLS (Transport Layer Security) — Layer 6 *(Optional)*
Standard Redis instances transmit plaintext across the network. However, production-grade clones often incorporate native SSL/TLS support to enforce confidentiality and prevent data tampering in transit.

### Direct Implications for Your Code:
* **Cryptographic Layering:** Instead of building cryptographic primitives from scratch, you will interface directly with libraries like **OpenSSL** or **BoringSSL**.
* **Socket Wrapping:** Your network architecture must wrap raw TCP file descriptors into a secure TLS context. Your code will handle the cryptographic TLS handshake immediately after the TCP connection registers, transparently decrypting incoming traffic before feeding it into your RESP parser, and encrypting outgoing bytes before passing them to the OS network interface.

## The Complete Pipeline inside Your Clone

When a client transmits a command to your database clone, the end-to-end processing pipeline traverses these protocols sequentially:

```text
[ Remote Client App ] 
         │
         │ (Encrypted Payload)
         ▼
  [ Layer 6: TLS ]       --> Decrypts the raw byte stream (if security is active)
         │
         │ (Plaintext Byte Stream: "*1\r\n$4\r\nPING\r\n")
         ▼
  [ Layer 4: TCP ]       --> Assembles out-of-order stream fragments
         │

         │                   of 1 Bulk String, and extracts the target token "PING".
         ▼
[ Core Database Engine ] --> Matches the token to your execution map and responds.
