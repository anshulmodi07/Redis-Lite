# Redis Clone V0.1 - Detailed Engineering Notes

## Overview

V0.1 is not primarily a feature release. It is an infrastructure and stability release that fixes foundational networking, protocol, and resource-management problems present in V0.

The primary objective of V0.1 is to transform a fragile TCP prototype into a reliable server foundation that can later support advanced Redis-like features such as RESP parsing, persistence, expiration, replication, and event-driven networking.

---

# Major Improvements

1. TCP Stream Handling (Line Buffering & Framing)
2. Resource Management & Lifetime Safety
3. Protocol Definitiveness & Error Handling
4. Architectural Trade-offs & Future Direction
5. Regression Testing Infrastructure

---

# 1. TCP Stream Handling

## The Fundamental TCP Misconception

V0 incorrectly assumed:

```text
One recv() = One Command
```

This assumption is false.

TCP is:

```text
Reliable Ordered Byte Stream
```

TCP is NOT:

```text
Packet Stream
```

The operating system may split or combine incoming data arbitrarily.

---

## Problem 1: Fragmentation

Client sends:

```text
PING\n
```

Network may deliver:

```text
PI
```

followed later by:

```text
NG\n
```

V0 behavior:

```text
recv() #1 -> "PI"
recv() #2 -> "NG\n"
```

Server processes:

```text
PI
```

and

```text
NG
```

as separate commands.

Result:

```text
ERR
ERR
```

even though the client sent a valid command.

---

## Problem 2: Coalescing

Client sends:

```text
PING\n
PING\n
```

Network may deliver:

```text
PING\nPING\n
```

in a single recv().

V0 processes:

```text
PING\nPING\n
```

as one command.

Result:

```text
ERR unknown command
```

---

# V0.1 Solution: Per-Client Receive Buffer

Each client connection receives its own input buffer.

Example:

```cpp
struct Client {
    int fd;
    std::string recvBuffer;
};
```

---

## Processing Flow

### Step 1

Receive bytes:

```cpp
recv()
```

Append to buffer:

```cpp
recvBuffer += incomingData;
```

---

### Step 2

Search for newline:

```cpp
recvBuffer.find('\n');
```

---

### Step 3

If newline exists:

Extract everything before newline.

Example:

```text
PING\n
```

Extract:

```text
PING
```

Execute command.

---

### Step 4

Remove processed data:

```cpp
recvBuffer.erase(0, pos + 1);
```

---

### Step 5

Continue processing until no complete command remains.

Pseudo-code:

```cpp
recvBuffer += recvData;

while (true) {
    auto pos = recvBuffer.find('\n');

    if (pos == std::string::npos)
        break;

    std::string cmd =
        recvBuffer.substr(0, pos);

    execute(cmd);

    recvBuffer.erase(0, pos + 1);
}
```

---

# Message Framing

TCP only delivers bytes.

Applications must define:

```text
Where does one message end?
```

This process is called:

```text
Message Framing
```

V0.1 uses:

```text
Delimiter-Based Framing
```

Delimiter:

```text
\n
```

---

# Oversized Input Protection

Potential attack:

```text
AAAAAAAAAAAAAAAAAAAA...
```

without newline.

Buffer grows indefinitely:

```text
1KB
10KB
100KB
1MB
100MB
...
```

Possible outcome:

```text
Out Of Memory
```

---

## Protection

Maximum buffer size:

```cpp
const size_t MAX_BUFFER = 8192;
```

Check:

```cpp
if(buffer.size() > MAX_BUFFER)
{
    send(...);
    close(...);
}
```

Response:

```text
ERR request too large
```

Connection terminated.

---

# Benefits

* Correct TCP behavior
* Handles packet fragmentation
* Handles packet coalescing
* Supports future pipelining
* Enables RESP implementation later
* Prevents memory exhaustion attacks

# 2. Resource Management & Lifetime Safety

## Problem A: Socket Rebinding

### Symptom

After stopping server:

```bash
Ctrl+C
```

Restart:

```bash
./server
```

Error:

```text
Address already in use
```

---

## Cause: TIME_WAIT

TCP maintains closed connections in:

```text
TIME_WAIT
```

state.

Purpose:

* Prevent delayed packets from corrupting future connections
* Ensure clean connection teardown

Typical duration:

```text
1–4 minutes
```

---

## Solution: SO_REUSEADDR

Before bind():

```cpp
int opt = 1;

setsockopt(
    server_fd,
    SOL_SOCKET,
    SO_REUSEADDR,
    &opt,
    sizeof(opt)
);
```

This tells the kernel:

```text
Allow immediate rebinding if only TIME_WAIT
entries are preventing reuse.
```

---

## Benefits

* Faster development
* Instant restart cycles
* Standard practice in production servers

---

# Problem B: Detached Thread Lifetime Hazards

V0 used:

```cpp
std::thread(...).detach();
```

for each client.

---

## What detach() Means

Detached thread runs independently.

Main thread no longer tracks it.

Diagram:

```text
Main
 └── Worker Thread
```

Worker may continue running after main changes state.

---

## Dangerous Design

Shared objects existed inside main():

```cpp
int main()
{
    std::unordered_map db;
    std::mutex db_mutex;
}
```

These objects live on the stack.

---

## Failure Scenario

Thread still running:

```cpp
db["key"] = "value";
```

Meanwhile:

```text
main() exits
```

Objects destroyed:

```text
db destroyed
mutex destroyed
```

Thread now accesses freed memory.

Possible results:

```text
Segmentation Fault
Memory Corruption
Random Crashes
Undefined Behavior
```

---

# V0.1 Fix

Move shared state to global scope.

```cpp
std::unordered_map<std::string,std::string> db;
std::mutex db_mutex;
```

---

## Memory Layout

Globals live in:

```text
Data Segment
BSS Segment
```

instead of stack memory.

Lifetime:

```text
Process Start
     ↓
Globals Exist
     ↓
Process End
```

Detached threads can safely access database during entire process lifetime.

---

# Benefits

* Prevents invalid references
* Eliminates major crash source
* Safe detached-thread access
* Stable shared state

# 3. Protocol Definitiveness

A protocol is a communication contract.

V0.1 introduces predictable and deterministic behavior.

---

# Symmetric Newlines

All commands end with:

```text
\n
```

All responses end with:

```text
\n
```

Examples:

```text
PING\n
```

Response:

```text
PONG\n
```

This allows both client and server to use identical line-buffering logic.

---

# CRLF Tolerance

Windows often sends:

```text
PING\r\n
```

instead of:

```text
PING\n
```

Without support:

```text
PING\r
```

fails string comparison.

---

## Solution

Remove carriage returns:

```cpp
'\r'
```

during parsing.

Example:

```text
PING\r
```

becomes:

```text
PING
```

---

# Deterministic Errors

Every invalid request maps to a known response.

---

## Unknown Command

Input:

```text
HELLO
```

Output:

```text
ERR unknown command
```

---

## Missing Arguments

Input:

```text
SET key
```

Output:

```text
ERR wrong number of arguments
```

---

## Oversized Request

Input:

```text
Huge buffer without newline
```

Output:

```text
ERR request too large
```

followed by disconnect.

---

# Why This Matters

Client libraries can reliably handle failures:

```python
if response.startswith("ERR"):
    handle_error()
```

Predictable behavior simplifies client development.

---

# New Command: PING

Request:

```text
PING
```

Response:

```text
PONG
```

---

## Purpose

Network heartbeat.

Checks:

* TCP connection alive
* Server running
* Parser functioning
* Thread/event loop responsive

---

## Characteristics

* No database access
* No writes
* No reads
* No side effects

Safe health-check command.

# 4. Architectural Trade-offs

V0.1 intentionally avoids scope creep.

The goal was stability, not feature expansion.

---

# Limitation 1: Single-Token Values

Currently:

```text
SET age 20
```

works.

But:

```text
SET greeting hello world
```

fails.

Reason:

Spaces are treated as argument separators.

Current parser likely uses:

```cpp
stringstream
```

tokenization.

---

## Future Solution

Options:

### Quoted Strings

```text
SET greeting "hello world"
```

### RESP

Length-prefixed values.

Planned for V0.2+.

---

# Limitation 2: Thread-Per-Client Architecture

Current model:

```text
1 Client = 1 Thread
```

Example:

```text
100 clients
=
100 threads
```

---

## Scaling Problems

Each thread requires:

* Stack memory
* Scheduler tracking
* Context switching

Large client counts cause:

```text
High RAM Usage
CPU Overhead
Reduced Scalability
```

---

## Future Direction

Event-driven networking:

Linux APIs:

```text
select()
poll()
epoll()
```

Redis primarily uses:

```text
epoll
```

---

# Why Not Now?

Changing to epoll requires:

* Rewriting networking layer
* Rewriting concurrency model
* Rewriting connection management

Too large for V0.1.

---

# Limitation 3: Plain Text Protocol

Current protocol:

```text
SET name devam
```

Easy to debug using:

```bash
nc localhost 6379
```

or

```bash
telnet localhost 6379
```

---

## Problems

Cannot safely handle:

* Spaces
* Newlines
* Binary data
* Images
* Arbitrary payloads

---

# Future: RESP

RESP = Redis Serialization Protocol

Example:

```text
*1
$4
PING
```

Response:

```text
+PONG
```

Advantages:

* Binary safe
* Supports spaces
* Supports large payloads
* Production-grade protocol
