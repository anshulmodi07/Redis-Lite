# TCP

## 1. Why TCP Matters

TCP (Transmission Control Protocol) is one of the most important protocols in networking because it provides:

* Connection-oriented communication
* Reliable data delivery
* Ordered delivery of data
* Error detection and recovery
* Flow control

For troubleshooting, TCP is often the first place to investigate:

### If TCP Looks Healthy

* No retransmissions
* No packet loss indicators
* No window issues
* No out-of-order packets

Then the issue is likely:

* Application layer
* Server-side processing
* Database/backend problems

### If TCP Looks Unhealthy

* Retransmissions
* Duplicate ACKs
* Out-of-order packets
* Zero-window situations

Then investigate:

* Network path
* Packet loss
* Congestion
* Routing issues

---

# 2. TCP Three-Way Handshake

The handshake establishes a reliable connection before data transfer begins.

## Steps

### Step 1: SYN

Client sends:

```text
SYN
Sequence Number = Client ISN
```

Purpose:

* Start connection
* Advertise client's Initial Sequence Number (ISN)

Example:

```text
Client ISN = 7413
```

---

### Step 2: SYN-ACK

Server responds:

```text
SYN + ACK
ACK = Client ISN + 1
Sequence Number = Server ISN
```

Example:

```text
ACK = 7414
Server ISN = 5106
```

This tells the client:

* Server exists
* Server received SYN
* Server is ready
* Server's starting sequence number is 5106

---

### Step 3: ACK

Client sends:

```text
ACK
ACK = Server ISN + 1
```

Example:

```text
ACK = 5107
```

Connection established.

---

## Why "SYN"?

SYN = Synchronize

The goal is to synchronize sequence numbers between both sides.

Both sides learn:

```text
Client starting sequence number
Server starting sequence number
```

---

# 3. Initial Sequence Numbers (ISN)

TCP chooses a large random starting number.

Example:

```text
Client: 177967413
Server: 205805106
```

These numbers:

* Must be unique
* Are randomly generated
* Prevent confusion between connections

---

## Wireshark Relative Sequence Numbers

Real numbers are huge.

Wireshark simplifies:

```text
Real Number → Relative Number
177967413 → 0
205805106 → 0
```

This makes analysis easier.

---

# 4. TCP Ports

A TCP connection is identified by:

```text
Source IP
Destination IP
Source Port
Destination Port
```

This combination is called a:

## Four Tuple

Example:

```text
10.1.1.5:54846
192.168.1.10:80
```

---

## Ephemeral Ports

Client usually uses a temporary port:

```text
54846
55900
60000
```

Characteristics:

* High-numbered ports
* Chosen dynamically
* Must be unique

---

## Well-Known Ports

Server usually listens on fixed ports:

Examples:

```text
80   HTTP
443  HTTPS
22   SSH
25   SMTP
```

# 5. Sequence Numbers

TCP reliability depends on sequence numbers.

Rule:

```text
1 Byte = 1 Sequence Number
```

---

## Example

Client sends:

```text
Sequence = 1
Payload = 501 bytes
```

Next sequence becomes:

```text
1 + 501 = 502
```

---

Server ACKs:

```text
ACK = 502
```

Meaning:

```text
I received all 501 bytes
```

---

# 6. Acknowledgment Numbers

ACK number tells:

```text
Next byte expected
```

Example:

Client sent:

```text
Bytes 1-501
```

Server replies:

```text
ACK = 502
```

Meaning:

```text
Received everything through byte 501.
Send byte 502 next.
```

---

# 7. Ghost Byte During Handshake

SYN consumes one sequence number even though no real data is sent.

Example:

```text
SYN sent with sequence 0
```

Server ACKs:

```text
ACK = 1
```

This incremented byte is called:

```text
Ghost Byte
```

---

# 8. TCP Reliability

When data is sent:

```text
Client stores copy in send buffer
```

Client waits for ACK.

If ACK arrives:

```text
Remove data from buffer
```

If ACK doesn't arrive:

```text
Wait for timer
Retransmit data
```

This mechanism creates reliability.

---

# 9. Round Trip Time (RTT)

RTT = Time for:

```text
Client → Server → Client
```

Measured using:

```text
SYN
SYN-ACK
```

Example:

```text
20 ms
```

Meaning:

Server is relatively close.

---

# 10. TTL Analysis

TTL helps estimate network distance.

Common starting TTL values:

```text
64
128
255
```

Example:

Received TTL:

```text
238
```

Likely:

```text
255 - 238 = 17 hops
```

Approximately 17 routers away.

# 11. TCP Flags

TCP uses control bits called flags.

---

## SYN

```text
Synchronize sequence numbers
```

Used during connection setup.

---

## ACK

```text
Acknowledgment
```

Used after handshake.

Almost every TCP packet contains ACK.

---

## FIN

```text
Graceful connection termination
```

---

## RST

```text
Immediate connection reset
```

---

## PSH (Push)

```text
Deliver data immediately to application
```

---

## Rarely Used

Usually ignored in basic troubleshooting:

```text
URG
ECE
CWR
NS
Reserved
```

# 12. TCP Window Size

Window size controls:

```text
How much data can be sent
before waiting for an ACK
```

Think of it as:

```text
Receiver's available buffer space
```

---

## Water Bucket Analogy

Sender = Water Source

Receiver Buffer = Bucket

Small bucket:

```text
Send
Wait
Send
Wait
Send
Wait
```

Very slow.

Large bucket:

```text
Send large amount
Continue sending
Higher throughput
```

---

# 13. Default Window Size Limitation

Original TCP:

```text
16-bit field
```

Maximum:

```text
65535 bytes
```

Because:

```text
2^16 - 1
```

---

## Problem

On high-latency links:

Example:

```text
UK ↔ California
```

Only sending:

```text
65535 bytes
```

before waiting for ACK

creates a bottleneck.

---

# 14. Window Scaling

Introduced to overcome 65535-byte limit.

Handshake advertises:

```text
Window Scale Option
```

Example:

```text
Shift Count = 6
```

Meaning:

```text
Multiply by 64
```

Actual window:

```text
65535 × 64
```

---

Another example:

```text
Shift Count = 9
```

Meaning:

```text
Multiply by 512
```

---

## Important

Window scaling is negotiated only during handshake.

If handshake isn't captured:

Wireshark cannot calculate:

```text
True Window Size
```

---

# 15. Dynamic Window Sizes

Window size changes continuously.

Example:

```text
131328
129856
131328
```

Normal behavior.

Reason:

* Data enters receive buffer
* Application consumes data
* Buffer frees space

---

## Bad Sign: Zero Window

```text
Window Size = 0
```

Meaning:

Receiver buffer is full.

Sender must stop sending.

This usually indicates:

* Application bottleneck
* Slow processing
* Resource exhaustion

---

# 16. TCP Options

TCP options are exchanged only during handshake.

This is why capturing the handshake is critical.

---

## MSS (Maximum Segment Size)

Defines maximum TCP payload per segment.

Example:

```text
1460 bytes
1440 bytes
```

Meaning:

```text
Do not send segments larger than this.
```

---

### Important

MSS is:

```text
Advertised
NOT negotiated
```

Values do not need to match.

---

## Window Scale Option

Enables large receive windows.

Example:

```text
Shift Count = 6
```

---

## SACK (Selective Acknowledgment)

Advertised during handshake.

Allows receiver to say:

```text
I received packets 1,2,4,5
but packet 3 is missing.
```

More efficient than retransmitting everything.

Requires support from both sides.

---

# 17. Why Capturing the Handshake is Important

Without handshake:

You lose:

* MSS
* Window Scale
* SACK capability
* Original sequence numbers
* Negotiated TCP features

Many troubleshooting mistakes happen because engineers start capturing after the connection already exists.

```text
Are bytes progressing correctly?
```

---

### Check ACKs

```text
Are acknowledgments arriving?
```

---

### Check Retransmissions

```text
Packet loss?
```

---

### Check Window Sizes

```text
Buffer bottleneck?
```

---

```text
Application problem?
```

# Key Takeaways

1. TCP reliability comes from Sequence Numbers and ACKs.
2. SYN means Synchronize Sequence Numbers.
3. Every byte increments the sequence number.
4. ACK = Next expected byte.
5. Handshake exchanges capabilities and options.
6. Window Size controls throughput.
7. Window Scaling solves the 65535-byte limitation.
8. MSS defines maximum payload per segment.
9. SACK improves recovery from packet loss.
10. Always capture the TCP handshake when troubleshooting.
